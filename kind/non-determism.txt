What can make scheduling non-deterministic
1) Concurrency & timing

Parallelism > 1: parallel Filter/Score runs race to produce the first “good enough” node; tiny timing shifts change outcomes.

Informer/cache timing: list/watch events arrive in different interleavings; the scheduler snapshot can differ across runs.

Backoffs & retries: exponential backoff timers fire at slightly different moments; queue ordering changes.

Timeouts (VolumeBinding, plugin timeouts): a call that barely times out in one run and not in another flips a decision.

2) Data ordering & tie-breaking

Go map iteration: if your plugin iterates over maps, order is randomized between processes.

Unstable set/heap ordering in plugins: any use of sets/heaps without explicit secondary sort keys will yield different ties.

Equal scores: when multiple nodes tie, the final tie-break can depend on internal iteration order unless you explicitly sort (by name) at the last step.

3) Default framework plugins that inject variability

PodTopologySpread (defaulting type = System): when a Pod doesn’t declare spread constraints, the system-applied rules can prefer different nodes based on current spread; tiny previous placements change the “best” target next time.

InterPodAffinity/AntiAffinity: matching depends on current pods; a small placement change early cascades.

ImageLocality: whether an image is present on a node (pull cache) can differ run-to-run.

NodeResourcesBalancedAllocation / LeastAllocated: resource rounding and near-ties can flip on small prior placements.

4) External cluster drift

Background controllers (descheduler, eviction manager) moving/evicting pods.

Node conditions/taints changing (e.g., brief memory pressure).

PV binding timing (VolumeBinding) and storage topology becoming ready at slightly different moments.

CNI / readiness jitter causing some nodes/pods to become schedulable a bit earlier/later.
